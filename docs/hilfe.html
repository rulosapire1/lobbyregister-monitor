1. Systemübersicht

Der Lobbyregister-Monitor ist ein automatisiertes System, das:

  • Täglich: Stellungnahmen und Gutachten aus dem Lobbyregister über die offizielle API V2 abruft und eine Übersichtsseite auf GitHub Pages aktualisiert
  • Montags: Eine wöchentliche Zusammenfassungs-Mail über Resend versendet
  • Montags: Einen automatischen Selbsttest durchführt und bei Problemen eine Admin-Mail sendet

4. Wie funktioniert der Datenabruf?

Das Skript nutzt die offizielle Lobbyregister API V2 (Swagger-Dokumentation) in zwei Schritten:

  1. Alle Registereinträge laden: Per /registerentries mit Cursor-Pagination werden alle ~6.800 Registereinträge geladen (dauert ca. 2–3 Minuten).
  2. Einzelabrufe und Filterung: Jeder Eintrag wird per /registerentries/{registerNumber} einzeln abgerufen und clientseitig gefiltert (dauert ca. 15–20 Minuten).

Filterkriterien

  • Themenfelder: Geprüft auf Ebene des Registereintrags unter activitiesAndInterests.fieldsOfInterest. Relevante Codes beginnen mit FOI_ENERGY_*, FOI_ENVIRONMENT_CLIMATE, FOI_EU_*, FOI_BUNDESTAG, FOI_ECONOMY_COMPETITION_LAW, FOI_POLITICAL_PARTIES, FOI_OTHER.
  • Empfänger: Geprüft auf Ebene jeder Stellungnahme unter statements[].recipientGroups[].recipients. Gesucht wird nach Ministerien mit „BMWE", „BMWK" oder „Wirtschaft" im Titel, sowie nach Bundestag-Einträgen.
  • Datum: Nur Stellungnahmen ab START_DATE (1. Januar 2026).

Beschreibungstexte

Die Inhaltsbeschreibung kommt aus regulatoryProjects[].description – das ist die Beschreibung des Regelungsvorhabens aus Sicht der jeweiligen Organisation. Verschiedene Organisationen können unterschiedliche Beschreibungen zum selben Vorhaben haben.

API-Änderungen erkennen

Der wöchentliche Selbsttest prüft automatisch, ob die API noch erreichbar ist und ob sich die API-Version (YAML-Datei) geändert hat. Bei Änderungen wird die Admin-E-Mail benachrichtigt. Die aktuelle API-Dokumentation ist unter api.lobbyregister.bundestag.de einsehbar.

5. Fehlersuche

Seite wurde nicht aktualisiert

  1. GitHub Actions öffnen: Link
  2. Den letzten Workflow-Durchlauf anklicken
  3. Den fehlgeschlagenen Job anklicken und die Logs lesen
  4. Häufige Ursachen: API-Key abgelaufen, API nicht erreichbar, GitHub Pages nicht aktiviert

Keine Stellungnahmen gefunden

  1. In den Logs prüfen: „Registereinträge geladen" – wenn 0, ist der API-Key ungültig
  2. Wenn Einträge geladen aber 0 Stellungnahmen: API-Antwortformat hat sich möglicherweise geändert
  3. Swagger-UI prüfen und Feldnamen in fetch_and_build.py vergleichen

Mail wurde nicht versendet

  1. In den GitHub Actions Logs den Job „Wöchentliche Mail versenden" prüfen
  2. Bei 403-Fehler: Resend-API-Key abgelaufen oder Empfängeradresse stimmt nicht mit Account-E-Mail überein
  3. Bei Resend unter resend.com → Logs prüfen

Manueller Start

Das System kann jederzeit manuell gestartet werden:

  1. GitHub Repository öffnen → Tab „Actions"
  2. Workflow „Lobbyregister Monitor" anklicken
  3. „Run workflow" → „Run workflow" klicken

6. System komplett neu aufsetzen (für Nachfolger)

Schritt 1: GitHub-Account

Falls kein Account vorhanden: github.com/signup – kostenlos registrieren.

Schritt 2: Repository anlegen

  1. Auf github.com einloggen → „+" → „New repository"
  2. Name: lobbyregister-monitor, Sichtbarkeit: Public, README anhaken
  3. „Create repository" klicken

Schritt 3: GitHub Pages aktivieren

  1. Repository → Settings → Pages
  2. Source: „GitHub Actions" auswählen → Speichern

Schritt 4: Secrets eintragen

Repository → Settings → Secrets and variables → Actions → folgende Secrets anlegen:

  • RESEND_API_KEY – von resend.com
  • EMAIL_RECIPIENT – Ziel-E-Mail (= Resend-Account-E-Mail)
  • ADMIN_EMAIL – Adresse für Fehlerbenachrichtigungen (= Resend-Account-E-Mail)
  • LOBBYREGISTER_API_KEY – Lobbyregister API-Key (öffentlich verfügbar auf der Infoseite)

Schritt 5: Dateien übertragen

Alle Dateien aus dem alten Repository in das neue kopieren. Am einfachsten:

  1. Altes Repo öffnen, jede Datei anklicken, „Raw" → Inhalt kopieren
  2. Im neuen Repo: „Add file" → „Create new file" → gleichen Pfad und Inhalt einfügen

Schritt 6: Resend-Account

Neuen kostenlosen Account auf resend.com erstellen und API-Key generieren. Wichtig: Im kostenlosen Plan kann nur an die E-Mail-Adresse gesendet werden, mit der der Account erstellt wurde.

Schritt 7: URLs anpassen

In folgenden Dateien den GitHub-Nutzernamen anpassen:

  • .github/workflows/update.yml – Zeile mit SITE_URL
  • scripts/fetch_and_build.py – Variable SITE_URL
