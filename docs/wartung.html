1. Systemübersicht

Der Lobbyregister-Monitor ist ein automatisiertes System, das täglich:

  • Stellungnahmen und Gutachten aus dem Lobbyregister des Deutschen Bundestages abruft
  • Einträge per KI (Google Gemini 3.1 Flash Lite) auf Energie-/Klimarelevanz prüft und Beschreibungstexte zusammenfasst
  • Eine öffentlich zugängliche Übersichtsseite auf GitHub Pages aktualisiert

Zusätzlich wird jeden Montag:

  • Eine wöchentliche Zusammenfassungs-Mail über Resend versendet
  • Ein Selbsttest (Health-Check) durchgeführt und bei Problemen eine Admin-Mail verschickt

Das System läuft vollautomatisch jeden Tag um 6:00 Uhr MESZ (= 4:00 Uhr UTC).

4. Ablauf eines täglichen Durchlaufs

Der GitHub Actions Workflow führt täglich folgende Schritte aus:

  1. Gemini-Cache laden – Bereits geprüfte Stellungnahmen werden aus dem GitHub Actions Cache wiederhergestellt, um API-Quota zu schonen.
  2. fetch_and_build.py – Daten vom Lobbyregister abrufen, nach Themenfeld und Empfänger filtern, data.json und initiale index.html generieren.
  3. gemini_enrich.py – Neue Stellungnahmen per Gemini auf Relevanz prüfen und Beschreibungen zusammenfassen. Ergebnisse werden im Cache gespeichert. Bereits gecachte Einträge werden übersprungen. Bei Quota-Erschöpfung: sofortiger Abbruch (Fail-Fast), ungeprüfte Einträge erhalten ein „Pending"-Badge. HTML wird mit aktuellen Daten neu generiert (inkl. Pending-Banner).
  4. GitHub Pages Deploy – Seite veröffentlichen
  5. send_email.py (nur montags) – Wochenmail versenden
  6. health_check.py (nur montags) – Selbsttest durchführen

5. Gefilterte Daten – was wird abgerufen?

Das Skript ruft alle Registereinträge über die Lobbyregister API v2 ab und filtert in zwei Stufen:

  • Pre-Filter (Orga-Ebene): Organisationen ohne jeglichen Energiebezug werden ausgeschlossen
  • Strikter Filter (Stellungnahme-Ebene): Nur Stellungnahmen mit relevanten Themenfeldern passieren den Filter. Orga-Felder dienen nur als Fallback.

Weitere Filterkriterien:

  • Empfänger: BMWE oder Bundestag
  • Themenfelder: Energie, Klimaschutz, EU-Binnenmarkt, EU-Gesetzgebung, Bundestag, Wettbewerbsrecht, Politisches Leben/Parteien, Sonstige
  • Zeitraum: ab 1. Januar 2026

Um die Filter zu ändern, müssen in scripts/fetch_and_build.py die Variablen FIELD_PRIORITY, TARGET_FIELD_CODES und START_DATE angepasst werden.

KI-Relevanzfilterung (Gemini)

Nach dem Datenabruf prüft gemini_enrich.py alle Einträge auf Bezug zum Aufgabenportfolio der Unterabteilung IIIA. Der Relevanzkatalog ist direkt im Skript als RELEVANZ_KATALOG hinterlegt und kann dort angepasst werden. Einträge der Priorität 1 (Energie/Wasserstoff) werden nie gefiltert.

Zusätzlich werden alle Beschreibungstexte per Gemini auf 2–5 Sätze zusammengefasst, mit fett markierten Schlüsselbegriffen. Die Originaltexte bleiben als original_summary in data.json erhalten.

Bereits geprüfte Einträge werden in docs/gemini_cache.json zwischengespeichert und beim nächsten Durchlauf nicht erneut an die API gesendet.

API-Änderungen erkennen

Der Selbsttest prüft wöchentlich automatisch, ob die Lobbyregister-API und die Gemini-API noch erreichbar sind. Bei Änderungen am API-Format wird die Admin-E-Mail benachrichtigt. Die aktuelle API-Dokumentation ist unter api.lobbyregister.bundestag.de einsehbar.

6. Fehlersuche

Seite wurde nicht aktualisiert

  1. GitHub Actions öffnen: Link
  2. Den letzten Workflow-Durchlauf anklicken
  3. Den fehlgeschlagenen Job (roter X) anklicken und die Logs lesen
  4. Häufige Ursachen: API-Key abgelaufen, Lobbyregister-API nicht erreichbar, GitHub Pages nicht aktiviert

Mail wurde nicht versendet

  1. In den GitHub Actions Logs den Job „Wöchentliche Mail versenden" prüfen
  2. Bei Resend unter resend.com → Logs prüfen ob Sendung registriert wurde
  3. Häufige Ursache: RESEND_API_KEY abgelaufen oder widerrufen → neuen Key erstellen und als Secret eintragen

Gemini-Anreicherung fehlgeschlagen

Erkennbar am gelben Warnhinweis auf der Webseite oder am „⏳ KI-Prüfung ausstehend"-Badge auf einzelnen Karten. Außerdem wird bei Quota-Erschöpfung eine Admin-Mail verschickt.

  1. In den GitHub Actions Logs den Step „Gemini-Anreicherung" prüfen
  2. Häufige Ursachen:
    • API-Key abgelaufen: Neuen Key auf aistudio.google.com/apikey erstellen (Account: lobbyregister.bot@gmail.com) und als GEMINI_API_KEY Secret eintragen
    • Quota erschöpft: Free Tier hat 500 req/Tag – bei vielen neuen Einträgen kann das Limit erreicht werden. Das Caching reduziert den Verbrauch erheblich. Ausstehende Prüfungen werden beim nächsten Durchlauf automatisch nachgeholt.
    • Rate Limit: Free Tier hat 15 req/min – das Skript wartet automatisch.
    • Modell nicht verfügbar: Google ändert gelegentlich Modellnamen. In gemini_enrich.py die Variable GEMINI_MODEL prüfen und ggf. anpassen. Aktuell: gemini-3.1-flash-lite-preview
  3. Der Monitor läuft auch ohne Gemini weiter – Einträge werden dann ungefiltert mit Originaltexten angezeigt

Manueller Start

Das System kann jederzeit manuell gestartet werden:

  1. GitHub Repository öffnen
  2. Tab „Actions" anklicken
  3. Workflow „Lobbyregister Monitor" anklicken
  4. „Run workflow" → „Run workflow" klicken

7. System komplett neu aufsetzen (für Nachfolger)

Falls das System auf einem neuen GitHub-Account neu aufgesetzt werden muss:

Schritt 1: GitHub-Account

Falls kein Account vorhanden: github.com/signup – kostenlos registrieren.

Schritt 2: Repository anlegen

  1. Auf github.com einloggen → „+" → „New repository"
  2. Name: lobbyregister-monitor, Sichtbarkeit: Public, README anhaken
  3. „Create repository" klicken

Schritt 3: GitHub Pages aktivieren

  1. Repository → Settings → Pages
  2. Source: „GitHub Actions" auswählen → Speichern

Schritt 4: Secrets eintragen

Repository → Settings → Secrets and variables → Actions → folgende Secrets anlegen:

  • LOBBYREGISTER_API_KEY – Lobbyregister API-Key (öffentlich auf der Infoseite verfügbar)
  • GEMINI_API_KEY – von aistudio.google.com/apikey (kostenlos, Account: lobbyregister.bot@gmail.com)
  • RESEND_API_KEY – von resend.com
  • EMAIL_RECIPIENT – Ziel-E-Mail-Adresse für die Wochenmail
  • ADMIN_EMAIL – Adresse für Fehlerbenachrichtigungen

Schritt 5: Dateien übertragen

Alle Dateien aus dem alten Repository in das neue kopieren. Am einfachsten:

  1. Altes Repo öffnen, jede Datei anklicken, „Raw" → Inhalt kopieren
  2. Im neuen Repo: „Add file" → „Create new file" → gleichen Pfad und Inhalt einfügen

Alternativ kann eine KI (z.B. Claude auf claude.ai) beim Aufsetzen helfen – einfach diese Wartungsdokumentation als Kontext einfügen und um Unterstützung bitten.

Schritt 6: Resend-Account

Neuen kostenlosen Account auf resend.com erstellen, API-Key generieren, Empfängeradresse verifizieren (unter „Audiences").

Schritt 7: Gemini API-Key

Google-Account erforderlich (lobbyregister.bot@gmail.com oder neuer Account). Auf aistudio.google.com/apikey einen kostenlosen API-Key erstellen. Der Free Tier reicht für den Monitor aus (15 Requests/Minute, 500 Requests/Tag).

Schritt 8: URLs anpassen

In folgenden Dateien den GitHub-Nutzernamen anpassen:

  • .github/workflows/update.yml – Zeile mit SITE_URL
  • scripts/fetch_and_build.py – Variable SITE_URL
  • scripts/template.html – Links in Header und Footer

Schritt 9: Relevanzkatalog anpassen (optional)

Der Katalog der relevanten Themen für die KI-Filterung ist in scripts/gemini_enrich.py in der Variable RELEVANZ_KATALOG hinterlegt. Er basiert auf dem Aufgabenportfolio der Unterabteilung IIIA. Bei organisatorischen Änderungen (z.B. neue Zuständigkeiten) sollte dieser Katalog angepasst werden.
